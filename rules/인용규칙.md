# 인용규칙 — 조문·판례 표기와 출처 핀

모든 서면·리서치 산출물의 법령·판례 인용은 이 규칙을 따른다.
**이 규칙의 존재 이유: 모든 인용을 기계 검증 가능하게 만드는 것.**

## 1. 법령 인용

- 본문 표기: `근로기준법 제27조`, `민법 제750조 제1항` (정식 법령명 + 조·항·호)
- 모든 법령 인용 직후에 출처 핀 병기:

  `[출처: 근로기준법 제27조 | MST:265959 | 시행 2025-10-23 [현행] | 조회 2026-07-03]`

  - 조회일은 MCP `get_law_text` 실행일. 상태 라벨은 MCP 반환 `[현행]`/`[연혁]`.
- 시행령·시행규칙은 별도 법령명(예: `근로기준법 시행령`)으로 조회해 핀에 함께 명시한다.
- 약칭(예: 주임법, 자배법)은 **검색용으로만** 사용한다(`rules/법령약칭.md` 참조).
  서면 본문과 출처 핀에는 항상 정식 법령명을 쓴다.

## 2. 판례 인용

- 본문 표기: `대법원 2022. 1. 14. 선고 2021두50642 판결` (법원명, 선고일, 사건번호, 판결/결정)
- 모든 판례 인용 직후에 출처 핀 병기:

  `[출처: 대법원 2022. 1. 14. 선고 2021두50642 판결 | 판례ID:222729 | 조회 2026-07-03]`

  - 판례ID는 MCP `get_decision_text`(`domain='precedent'`) 반환 안정 ID.
- **생사확인 라인.** 리서치 파일(`리서치/쟁점별_법리.md`)에서는 판례 출처 핀 바로 아래에
  MCP `legal_analysis`(`mode='cite_check'`) 역인용 추적 결과를 한 줄 기재한다:

  `[cite_check: 2026-07-03 조회 — 역인용 추적상 변경·폐기 미발견]`

  변경·폐기 정황 발견 시:

  `[⚠️ cite_check: 대법원 YYYY. M. D. 선고 ○○○○ 전원합의체 판결이 변경 선언 — 변호사 확인 필요]`

  - `legal_analysis`(`mode='cite_check'`) 결과는 역인용 추적 기반 판정이며,
    판례 폐기 여부의 최종 판단은 변호사가 한다. 미발견이 "현재 유효"를
    보증하지 않는다.
  - `cite_check` 라인은 리서치 파일 전용 표기로, 서면 본문에는 넣지 않는다.

## 3. 절대 규칙

1. **원문 발췌 없는 인용 금지.** 리서치 파일(`리서치/쟁점별_법리.md`)에 원문 발췌가
   없는 조문·판례는 어느 서면에도 인용할 수 없다.
2. **출처 핀 없는 법률 주장 금지.** 조문·판례를 언급하는 모든 문장에는 핀이 있어야
   하며, /verify-citations가 핀 없는 법률 주장을 ❌ 처리한다.
3. 출처 핀은 **초안 DOCX까지 유지**한다(검증 추적성). 최종 제출본에서 변호사가 제거한다.
   - 초안 DOCX에서 `[출처: …]`는 **빨간색**, `[변호사 확인 필요…]`는 **파란색**으로
     자동 표시된다(render_서면.py 후처리) — 변호사 검수 시 제거·확정 지점 식별용.
4. 수치·이율·기간 등 법정 값은 조문 원문에서 확인된 것만 쓴다. 미확인이면
   `[변호사 확인 필요]`를 본문에 표기하고 사건컨텍스트 `확인필요`에 적재한다.
5. **출처 핀에는 안정 ID(MST/판례ID)만 쓴다.** MCP 응답의 `링크` 필드 등
   원시 링크 문자열(API 키 포함)은 리서치·서면·검증보고 어떤 파일에도
   넣지 않는다(API 키 노출 방지).

## 4. 조회 범위 한계 고지

리서치 산출물 말미에 다음 문구를 고정 기재한다:

> 본 리서치는 korean-law-mcp(국가법령정보센터) 조회 결과이며 조회일 기준임.
> 미공개·하급심 판례 등 조회 범위 밖은 별도 확인 필요.

## 5. 생사확인 표준 절차 (단일 출처 — 스킬은 이 절을 참조한다)

1. korean-law-mcp `legal_analysis`(`mode='cite_check'`, `caseNumber`, 필요시
   `deepScan`)에 대상 판례 사건번호를 넣어 역인용 추적 결과를 받는다.
1-b. **결과의 `대상:` 사건번호가 질의 사건번호와 정확히 일치하는지 반드시
   확인한다.** 질의 번호가 DB에 정확히 없으면 무관한 판례를 조용히 반환하는
   prefix 오매칭이 실측됨(예: `2020도68` 질의 → `2020도6874` 반환, 경고 없음).
   불일치면 그 결과는 무효 — 사건번호를 재확인하거나
   `search_decisions`(`domain='precedent'`) 역검색으로 올바른 판례를 특정한 뒤
   재실행한다.
2. 변경·폐기 선언(전원합의체 변경, 저촉 범위 폐기 등)이 있으면 리서치는
   후속 판례로 교체, 검증은 ❌. 단순 "등 참조" 원용은 변경이 아니다.
3. 결과를 §2의 생사확인 라인 형식으로 기재한다.

수행 주체별 차이:
- /legal-research: 변경·폐기 발견 시 후속 판례(현재 법리)로 교체.
  불명확하면 발췌 유지 + ⚠️ 병기.
- /verify-citations: 리서치의 생사확인 라인을 신뢰하지 말고
  `legal_analysis`(`mode='cite_check'`) 재실행(검증자 독립성).

## 6. MCP 검색 규칙

- 법령: `search_law`로 정식 법령명·약칭 양쪽 검색 후 `get_law_text`로 원문 확인.
- 판례: `search_decisions`(`domain='precedent'`)로 쟁점 키워드 검색,
  `get_decision_text`(`domain='precedent'`)로 원문 확인.
- 민사 외 사건종류(일반행정 등)는 `search_decisions`(`domain`으로 18개 도메인
  통합 검색)로 쟁점에 따라 검색.
- 신규 도메인(행정규칙·조약·해석례 등)은 쟁점 성격이 맞을 때만
  `search_decisions`(`domain='X'`)/`get_decision_text`(`domain='X'`)를 추가한다
  (도메인 enum과 판단 기준은 legal-research 스킬). 자치법규 전용 도메인은
  없으므로 자치법규 비교는 `search_law` 계열 또는 `execute_tool` 프록시로
  수행한다.
- 결과가 많으면 대법원·현행 우선으로 좁힌다.

## 7. 행위시법 조회 표준 절차 (단일 출처 — 스킬은 이 절을 참조한다)

인용해야 하는 것은 기준일에 시행 중이던 조문이다.

기준일: 행위시점(계약 체결일·불법행위일·해고일 등)을 사건컨텍스트.사실관계에서
특정. 불명확하면 [변호사 확인 필요].

절차:
1. `legal_analysis`(`mode='applicable_law'`, `lawName`, `date`, 필요시 `jo`)에
   법령·조문·기준일을 넣어 기준일 시행 버전과 경과규정을 받는다.
2. 반환된 버전의 MST·시행일자를 확인한다.
3. 현행(get_law_text 최신)과 대조:
   - 동일 → 현행 핀 + 부기 [행위시 확인: 기준일 YYYY-MM-DD 시행 버전과 조문 동일]
   - 차이 → 행위시 버전 발췌 + 연혁 핀. 경과규정(부칙 적용례)을 확인·발췌.
     적용 버전 최종 판단은 [변호사 확인 필요 — 적용 법령 버전].

연혁 핀 표기:
[출처: 근로기준법 제27조 | MST:{연혁MST} | 시행 YYYY-MM-DD [연혁] | 기준일 YYYY-MM-DD | 조회 YYYY-MM-DD]

- **부기의 지위**: `[행위시 확인: …]` 부기는 출처 핀 블록의 일부다.
  리서치의 핀을 서면에 인용할 때 부기도 함께 복사한다(이관 누락 시
  /verify-citations가 ⚠️ 처리). `[cite_check: …]` 라인은 종전대로 리서치
  파일 전용이며 서면에 넣지 않는다(§2).
- 검증 재현: `legal_analysis`(`mode='applicable_law'`, `lawName`, `date`, 필요시
  `jo`) 재실행, 또는 `get_law_text`(`mst`, 필요시 `jo`)에 **핀의 시행일을
  `efYd`(YYYYMMDD)로 병행**해 같은 버전을 재현한다(기계 검증 가능성 유지).
  - 연혁 MST는 `efYd` 없이 단독 조회하면 `NOT_FOUND`가 반환되는 사례가
    실측됨(예: MST:279829 단독 → NOT_FOUND, `efYd=20251111` 병행 → 정상).
    단독 조회 실패를 곧바로 "조문 부존재"로 판정하지 말 것.
